Sección: Estructura organizacional: Herencia y roles
Contenido: KontrolTravel usa una estructura multinivel: una configuración en el nivel más alto (padre) aplica a todos los niveles inferiores. Sección: Herencia de configuraciones
Contenido: El modelo se basa en la herencia de configuraciones. Si un acuerdo o una política se establece en la Entidad Padre, se aplica a todas las Entidades Hijas por defecto. Esto minimiza el esfuerzo de configuración y garantiza la coherencia global. Solo se configuran de forma local aquellos parámetros que deben ser específicos o diferentes en la entidad hija. En conclusión: Entidad Padre: Establece la configuración base (políticas, parámetros). Entidades Hijas: Heredan automáticamente las configuraciones de la Entidad Padre. Los ajustes pueden ser sobrescritos a nivel de la Entidad Hija si se requiere una configuración más específica. Herencia Multinivel. Al ser una estructura de N niveles, una configuración en nivel mas alto puede aplicar a todos los niveles inferiores, el comportamiento en una entidad inferior es cambiado solo cuando se sobre-escribe la configuración para una entidad especifica indicando que no asuma la configuración que exista en una entidad padre. Documento: FlujoCorporativoKT
Sección: Tipos de Flujo de Aprobación
Contenido: KontrolTravel soporta tres tipos de flujos de aprobación para adaptarse a las necesidades de control de cada cliente: Flujo por Defecto o Cualquiera Aprueba: La solicitud queda aprobada tan pronto como cualquiera de los aprobadores definidos (en cualquier nivel) dé su visto bueno. Flujo Paralelo: Todos los aprobadores definidos deben aprobar la solicitud. El orden en que aprueban no es relevante; el sistema solo avanza a la emisión cuando la totalidad de los aprobadores ha dado su consentimiento. Flujo Secuencial: La aprobación se realiza en serie, respetando estrictamente el orden predefinido. Tan pronto como el primer aprobador aprueba, la notificación se envía al segundo; cuando el segundo aprueba, se envía al tercero, y así sucesivamente hasta que el último aprobador finaliza el flujo.

Contenido: Los UDID (Identificadores Únicos) son campos que permiten personalizar las necesidades específicas de un cliente. Todo campo gerencial (UDID) debe ser creado con un nombre que no contenga signos (*) ni espacios, y debe tener un alias (nombre que se verá en el proceso de reserva). Un campo que no cumpla con estas características no podrá ser guardado.

Documento: SPLIT LATAM
Sección: Introducción.
Contenido: Con el objetivo de optimizar la gestión de reservas y brindar mayor flexibilidad operativa a las agencias de viaje, se pone a disposición la funcionalidad de SPLIT de pasajeros para reservas LATAM vía NDC. Esta funcionalidad permite dividir pasajeros pertenecientes a una reserva ya emitida, generando los registros, órdenes y documentos correspondientes para cada nueva reserva resultante, siempre en cumplimiento de las políticas y restricciones definidas por el proveedor. El presente documento tiene como finalidad describir el alcance de la funcionalidad de SPLIT, así como detallar las consideraciones operativas que deben evaluarse previo a su utilización y los permisos de usuario requeridos para acceder y ejecutar este proceso correctamente.
URL documentación: https://pacardenaz.github.io/parametros-manual/docs/10_split-latam.html

Documento: SPLIT LATAM
Sección: Funcionalidad SPLIT
Contenido: La funcionalidad SPLIT LATAM permite dividir uno o más pasajeros de una reserva ya emitida, generando una nueva orden y un nuevo localizador interno (LOC) en Kontrol, asociado al registro creado a partir del SPLIT realizado. Una vez emitida la reserva, se podrá realizar este Split seleccionando la opción “Dividir PNR”. El sistema desplegara una nueva ventana en la que se visualizarán los nombres de todos los pasajeros asociados a la reserva. Desde esta ventana, la agencia deberá seleccionar los pasajeros que desea dividir de la reserva original para continuar con el proceso.
URL documentación: https://pacardenaz.github.io/parametros-manual/docs/10_split-latam.html

Documento: SPLIT LATAM
Sección: Consideraciones importantes antes de realizar un SPLIT
Contenido: El SPLIT no es reversible Una vez realizada la división, no es posible anular la nueva orden generada desde el sistema. Si la agencia requiere la anulación de los tiquetes divididos (nueva orden), deberá gestionarse directamente con LATAM. Consideraciones operativas Teniendo en cuenta el anterior punto, en escenarios donde una reserva inicial contiene uno o más pasajeros, y posteriormente se identifica que solo alguno de ellos realizara el viaje, mientras que otros deben ser cancelados se recomienda aplicar el siguiente procedimiento: Ejecutar el Split (nueva orden) de los pasajeros que, si van a realizar el viaje, de manera que estos queden asociados a nuevos localizadores y a nuevas reservas. Mantener en la reserva original a los pasajeros que no realizarán el viaje, con el fin de que su cancelación pueda efectuarse directamente desde la herramienta, sin necesidad de gestionar el proceso con LATAM.
URL documentación: https://pacardenaz.github.io/parametros-manual/docs/10_split-latam.html
